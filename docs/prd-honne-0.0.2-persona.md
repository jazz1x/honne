## 1. Problem Definition

### 1.1 Current State (As-Is)

`/honne:whoami` produces a read-only mirror: `persona.json` and `docs/honne.md` show who the user is, but the output is passive. The user cannot *use* the persona — it cannot speak, respond, or reflect back. The antipattern axis captures inefficiency patterns; the signature axis captures what consistently works. Both exist in the same file, but there is no mechanism to put them in tension and produce a working verdict.

Current state:
- `persona.json` holds 7 axes including `antipattern` and `signature` with claims, explanations, and quotes
- No skill turns these into an active prompt
- The "coming soon" teaser in `/honne:whoami` completion step points to this feature explicitly

### 1.2 Target State (To-Be)

`/honne:persona` invoked in any Claude Code session:
1. Reads `.honne/persona.json`
2. Surfaces the `antipattern` and `signature` axes as opposing forces
3. Synthesizes a **conflict resolution verdict** — who this person is when the two fight
4. Generates a **system prompt** that lets Claude respond as this person
5. Activates it in the current session by outputting the system prompt as skill output text — Claude honors it for the remainder of the session. **Session-scoped only: does not persist to the next session.** CLAUDE.md injection is prohibited.
6. Saves it to `.honne/persona-prompt.md` for reuse outside Claude Code (paste into any session or LLM)

Success: the user can call `/honne:persona`, immediately converse with a Claude that mirrors their working patterns for that session, and take the generated prompt file anywhere for future sessions.

### 1.3 Business/Technical Impact

Without this:
- `persona.json` is a report, not a tool — passive data with no downstream use
- The `signature` axis (added in 0.0.1) has no consumer; its value is unrealized
- The antipattern × signature pairing — designed as counterparts — has no synthesis step

With this:
- honne closes the loop: observe → synthesize → embody
- The persona prompt becomes a portable artifact (usable in non-Claude-Code environments)
- Sets the foundation for 0.0.3 (token analysis guide could be framed *through* the persona)

## 2. Design Decision

### Alternative A: Prompt Export (Static File)

- **Approach**: Render persona.json into a markdown system prompt file. User copies it manually into sessions.
- **Pros**: Zero session coupling. Works with any LLM. Simple to implement.
- **Cons**: No in-session activation. No conflict synthesis. Feels like a report export, not a "分身術."
- **Suitable situation**: When portability across LLM providers is the primary goal.

### Alternative B: In-session Skill Activation (Ephemeral Clone)

- **Approach**: `/honne:persona` skill reads persona.json and instructs Claude to respond as the user persona for the rest of the session. Ephemeral — no file written.
- **Pros**: Immediate activation. "分身術" feel. No CLAUDE.md injection (guardrail compliant).
- **Cons**: No export. Persona quality depends entirely on prompt engineering in the skill. No explicit conflict synthesis step.
- **Suitable situation**: When immediate in-session use is the only goal.

### Alternative C: Conflict Synthesis + Dual Export (Selected)

- **Approach**: `/honne:persona` extracts `antipattern` and `signature` claims, runs a LLM synthesis step (modeled on `whoami` Step 5) to resolve the conflict into a verdict, then both activates the persona in-session and saves `.honne/persona-prompt.md`.

```
[antipattern]  overspec(251), repeat_same_request(82)
      ↕  conflict
[signature]    decisive_close, targeted_request
      ↓  synthesis (LLM call via persona_synthesis_prompt template)
"빠르게 결론 내리지만 초기 설계에 과잉 투자하는 사람 — 단정하되 첫 판단을 너무 믿지 말 것"
      ↓
Claude-as-you system prompt → in-session activation + .honne/persona-prompt.md
```

- **Pros**: Directly satisfies "겁나 싸워서 결론주게끔." Both axes get a consumer. Produces portable artifact. Mirrors the synthesis pattern already proven in `whoami`.
- **Cons**: Largest implementation scope. Synthesis prompt quality determines output quality.
- **Suitable situation**: When the goal is a living, tension-aware persona — not just a static profile dump.

### Selection: Alternative C

- **Rationale**: `antipattern` and `signature` were designed as counterparts (0.0.1 explicit decision). The synthesis pattern (LLM call via locale template) is already proven in `whoami` Step 5 — reuse cost is low. A and B fail the "분신술처럼" + "싸워서 결론주게끔" requirement.
- **Rejected**: A (no conflict synthesis, no in-session activation), B (no export, no explicit conflict step).
- **Validity conditions**: Revisit if both `antipattern` and `signature` are `null` in a user's persona.json — in that case, fall back to Alt B (activate with whatever axes are present, skip conflict synthesis).
- **Referenced existing assets**: `[guardrail] shim bash는 3줄 이하, 로직은 Python에만` — applies to CLI entry. `[pattern] honne 리팩터 구현 패턴` — persona_prompt module follows same package structure.

## 3. Technical Design

### 3.1 Architecture Overview

```
/honne:persona (SKILL.md)
      │
      ├── Step 1: Read .honne/persona.json
      │
      ├── Step 2: Build CONFLICT_PAYLOAD
      │     antipattern.claim + antipattern.explanation
      │     signature.claim   + signature.explanation
      │     (null axis → mark as absent, skip that side of conflict)
      │
      ├── Step 3: LLM synthesis (Claude mental reasoning)
      │     system: skills/persona/templates/persona_synthesis_prompt.{locale}.md
      │     user:   CONFLICT_PAYLOAD (JSON)
      │     output: { verdict, character_oneliner, system_prompt } (strict JSON)
      │
      ├── Step 4: Save
      │     Write → {PWD}/.honne/persona-prompt.md   (portable export)
      │     Write → {PWD}/.honne/cache/persona-synthesis.json  (raw synthesis output)
      │
      └── Step 5: In-session activation
            Output the system_prompt to the user with activation instructions
            ("Copy this to your session system prompt, or use it as context below.")
```

New CLI subcommand:
```
honne render persona-prompt \
  --persona .honne/persona.json \
  --synthesis .honne/cache/persona-synthesis.json \
  --locale ko \
  --out .honne/persona-prompt.md
```

### 3.2 Core Data Model

```python
# CONFLICT_PAYLOAD sent to LLM synthesis
@dataclass
class ConflictPayload:
    locale: str                          # "ko" | "en" | "jp"
    antipattern: AxisSummary | None      # None if insufficient_evidence
    signature: AxisSummary | None        # None if insufficient_evidence
    supporting_axes: dict[str, AxisSummary]  # remaining 5 axes as context

@dataclass
class AxisSummary:
    claim: str
    explanation: str
    evidence_strength: float

# LLM synthesis output (strict JSON)
@dataclass
class PersonaSynthesis:
    verdict: str            # 1–2 sentence conflict resolution
    character_oneliner: str # ≤20 words, who this person IS
    system_prompt: str      # full system prompt for Claude-as-user
    conflict_present: bool  # False if one side was null
```

```python
# .honne/persona-prompt.md structure
"""
# [character_oneliner]

## Who you are
[verdict]

## Behavioral signature
[signature.claim if present]

## Watch out for
[antipattern.claim if present]

## System prompt
---
[system_prompt]
---
"""
```

## 5. Edge Cases & Error Handling

| Scenario | Expected Behavior | Handling |
|----------|------------------|----------|
| `persona.json` missing | Exit 66, tell user to run `/honne:whoami` first | SKILL Step 2 check |
| Both `antipattern` and `signature` are `null` | `conflict_present: false` — synthesis prompt enters "portrait mode": describe the person from the 5 remaining axes only, no conflict framing. `verdict` = single-axis portrait, `character_oneliner` still required, `system_prompt` generated from available axes | Synthesis prompt must include explicit branch: `if conflict_present is false → skip conflict framing, describe from supporting_axes` |
| Only one side null (e.g., signature present, antipattern null) | `conflict_present: false` — one-sided synthesis. Present the existing axis as the dominant trait; note the absent axis as "not yet observed" in `verdict`. Do not fabricate the missing side. | Synthesis prompt branch: `if one side null → dominant-trait portrait, label absent side as unobserved` |
| `persona.json` stale (run_ts > 7 days) | Warn user "persona last updated N days ago — consider re-running `/honne:whoami`" | SKILL Step 2 staleness check |
| LLM synthesis returns malformed JSON | Skip saving `persona-synthesis.json`; output raw synthesis text with warning | Same pattern as whoami Step 5 narrative fallback |
| `system_prompt` field empty in synthesis | Re-render from template using available fields; skip activation | Fallback render path in `render_persona_prompt` |
| User runs `/honne:persona` before any `whoami` run | Exit 66, clear error | SKILL Step 2 |
| Locale mismatch (persona generated in `ko`, skill invoked in `en`) | Use invoked locale for synthesis prompt; persona data language is irrelevant to template | No special handling needed |

## 7. Guardrails

### Project Guardrails (soft — warn on violation)

- `persona-prompt.md` must not contain raw quotes from `persona.json` verbatim — synthesis should transform, not copy
- `persona_synthesis_prompt` templates must include explicit instruction: "do not invent behavioral claims not present in the input axes"
- Staleness threshold defaults to 7 days; override via env var `HONNE_PERSONA_STALE_DAYS=<int>`
- `system_prompt` in synthesis output must be ≤ 1500 tokens (guard against runaway generation)

### Prohibitions (hard — stop immediately on violation)

- **No CLAUDE.md injection** — persona activation is in-session only via skill output; never write to CLAUDE.md (self-reinforcement loop guard, existing guardrail)
- **No network calls** — synthesis is Claude's own mental reasoning; no external API calls
- **No heredocs in SKILL.md bash blocks** — follows 0.0.1 SKILL.md hardening
- **No `/tmp` writes** — all output paths are explicit `{PWD}/.honne/...`
- **bash shim ≤ 3 lines** — all logic in `honne_py/persona_prompt.py`, not in shell
- **`scan.json` schema must not be touched** — persona feature reads `persona.json` only
